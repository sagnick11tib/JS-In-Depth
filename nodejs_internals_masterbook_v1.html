Part A · Foundation

Core Mental Model: What Node.js Really Is

Node.js is not just JavaScript. It is JavaScript + V8 + Node core APIs + C++ bindings + libuv + the operating system.

01
Easy memory model
Node.js Runtime
=
JavaScript Code
+ V8 Engine
+ Node Core APIs
+ Native C++ Bindings
+ libuv
+ Operating System
🧠

JavaScript layer

Your application code, npm packages, callbacks, promises, async functions, classes and modules.

⚙️

Runtime layer

Node exposes APIs like fs, http, crypto, stream, timers and process.

🧩

Native layer

Node connects JavaScript to lower-level C/C++ systems, libuv, OpenSSL, zlib and the OS kernel.

✅

Interview shortcut: JavaScript in Node is single-threaded by default, but the Node runtime is not just one thread. Async work can be delegated to the OS, libuv, or the libuv thread pool.

🎤 60-second answer

Node.js is a JavaScript runtime built on V8. V8 executes JavaScript and manages memory. Node adds server-side APIs such as filesystem, networking, crypto, streams, process and timers. Many of those APIs are implemented through Node core and native C++ bindings, then delegated to libuv and the operating system. So your JavaScript runs on one main execution thread by default, but Node uses OS async I/O and a worker thread pool to avoid blocking the event loop.

Ignition

V8's interpreter executes bytecode. It helps code start quickly before heavy optimization.

Optimizes hot code paths. If assumptions become invalid, V8 can deoptimize.

Object shape example
function createUser(name, age) {
  return { name, age };

const b = createUser('Ravi', 28);

// Better: same property order, stable shape.
// Worse: adding random properties later to each object.

Every output prediction question starts with the call stack. Every memory question starts with the heap and references.

05

Tracks currently running function calls. If it is busy, Node cannot process another JS callback.

Stores objects, arrays, functions, closures and long-lived data. Memory leaks happen when data remains reachable.

Contains lexical environment, variable environment, scope chain and this.

Part B · V8

Garbage Collection & Memory in Node.js

Memory leaks are not objects that cannot be freed. They are objects your program accidentally keeps reachable.

06
V8 heap model
V8 Heap
|
├── Young Generation
|   └── Short-lived objects
|
├── Old Generation
|   └── Long-lived objects
|
└── External / Native Memory
    ├── Buffers
    ├── Native addons
    └── Some C++ allocations

Minor GC

Collects short-lived objects in the young generation. Usually frequent and relatively fast.

Major GC

Collects old generation memory. Can be more expensive and can cause pauses.

Leaks

Common causes include global maps, unbounded caches, event listeners, timers, closures and retained request objects.

Simple leak pattern
const cache = new Map();

function handleUser(userId, profile) {
  // Problem: no eviction strategy.
  cache.set(userId, profile);
}
Remember: GC can only free unreachable memory. If your app still references it, it stays.

🎤 Interview short

Node memory is mainly managed by V8's garbage collector, but Buffers and native modules can also involve external memory. A memory leak happens when data remains reachable accidentally, for example through global caches, listeners, timers or closures. I would debug it with process.memoryUsage(), heap snapshots, allocation timelines and checking retained references.

Part D · libuv

Timers Phase: setTimeout & setInterval

A timer says “not before this time”, not “exactly at this time”.

11
Timer delay is a threshold
setTimeout(() => {
  console.log('timer callback');
}, 0);

// Heavy sync work delays the timer.
const start = Date.now();
while (Date.now() - start < 100) {}

console.log('sync done');

Threshold

setTimeout(fn, 0) means run after at least 0ms and when the event loop can reach the timer callback.

Blocked loop

If the call stack is busy, timer callbacks wait.

Interval caution

setInterval can pile pressure if the callback work takes too long.

Remember: Timer delay is a minimum threshold. The event loop must be free enough to run it.
Part D · libuv

Poll Phase: I/O Heart of Node

Poll is where Node receives I/O callbacks and may wait for new I/O.

12
Poll phase mental model
Poll Phase
|
├── Check completed I/O
├── Execute I/O callbacks
├── Wait for new I/O if appropriate
└── Move to check phase if setImmediate is queued
I/O callback example
const fs = require('node:fs');

fs.readFile(__filename, () => {
  console.log('file read callback');
});

Network callbacks

Socket and HTTP callbacks are commonly delivered through I/O readiness mechanisms.

File callbacks

Many filesystem operations complete through the libuv worker pool, then their callback is scheduled back.

🎤 Interview short

The poll phase is central to Node because it handles many I/O callbacks and can wait for new I/O events. If the poll queue has callbacks, Node executes them. If not, Node may wait, move to timers if timers are ready, or move to the check phase if setImmediate callbacks are queued.

Part D · libuv

Check Phase: setImmediate

setImmediate is designed to run after the poll phase.

13
setImmediate vs setTimeout outside I/O
setTimeout(() => console.log('timeout'), 0);
setImmediate(() => console.log('immediate'));

// Outside I/O, order can be timing-dependent.
Inside I/O callback
const fs = require('node:fs');

fs.readFile(__filename, () => {
  setTimeout(() => console.log('timeout'), 0);
  setImmediate(() => console.log('immediate'));
});

// Commonly: immediate before timeout inside I/O.
🧭

Remember: setImmediate belongs to the check phase. Inside an I/O callback, the event loop is around poll, so check usually comes before the next timers phase.

Part E · Queues

process.nextTick Queue

nextTick is powerful and dangerous. It runs before promise microtasks in CommonJS-style scheduling.

14
nextTick position
Current JS callback finishes
        |
        v
process.nextTick queue
        |
        v
Promise microtask queue
        |
        v
Continue event loop phases
nextTick vs promise
Promise.resolve().then(() => console.log('promise'));
process.nextTick(() => console.log('nextTick'));
console.log('sync');

// CommonJS expected: sync, nextTick, promise
⚠️

Starvation risk: Recursively scheduling process.nextTick can prevent Node from reaching I/O phases.

🎤 Interview short

process.nextTick schedules a callback to run immediately after the current operation finishes and before Node continues the event loop. It is useful for APIs that must emit asynchronously but very soon. But overusing or recursively scheduling it can starve I/O because Node keeps draining nextTick callbacks before moving forward.

Part E · Queues

Promise Microtasks & async/await

Promises and await continuations are microtasks. They run after nextTick but before normal event loop callbacks in typical CommonJS ordering.

15

Promise.then

Schedules a reaction into the microtask queue when the promise settles.

queueMicrotask

Directly queues a microtask using the same general microtask mechanism used by promise handlers.

await

Suspends the async function and resumes the continuation as a microtask after the awaited value settles.

async/await order
async function run() {
  console.log('A');
  await null;
  console.log('B');
}

run();
console.log('C');

// A, C, B
Remember: await pauses the function, not the Node process.

Full Node Scheduling Priority

For output questions, this is your scoring formula.

16

1. Current synchronous JavaScript
2. process.nextTick queue

Classic interview output
setTimeout(() => console.log('timeout'), 0);
setImmediate(() => console.log('immediate'));
Promise.resolve().then(() => console.log('promise'));
process.nextTick(() => console.log('nextTick'));
console.log('sync');
✅

Safe answer: sync first, then nextTick, then promise. The order between timeout and immediate can be environment/timing dependent outside I/O. Inside I/O, setImmediate commonly runs before the timer.

🎤 Interview short

My first pass is synchronous code. Then I drain Node's nextTick queue, then promise microtasks. After that, Node continues through libuv phases, so timers, I/O poll callbacks, setImmediate and close callbacks run depending on the current phase and readiness.

Part F · Async Work

libuv Thread Pool

The thread pool is why Node can run some expensive operations away from the event loop thread, but it can also become a bottleneck.

17

        |
        v

        |

        |

Uses thread pool

Many fs, crypto, zlib and dns.lookup operations can use the pool.

Default size

The libuv thread pool has a default size. It can be configured with UV_THREADPOOL_SIZE before the process starts.

Saturation

Heavy crypto or compression can occupy workers and delay unrelated filesystem operations.

Thread pool pressure
const crypto = require('node:crypto');

for (let i = 0; i < 8; i++) {
  crypto.pbkdf2('secret', 'salt', 500000, 64, 'sha512', () => {
    console.log('crypto done', i);
  });
}
Remember: Event loop free does not mean all async work is free. The worker pool can be full.
Part F · Async Work

Network I/O vs File I/O

Network and file I/O are both async from your JS point of view, but internally they are often handled differently.

18

Network I/O

  • Often uses OS readiness notifications
  • Good fit for event loop model
  • Examples: TCP, HTTP, sockets
  • Usually not consuming libuv thread pool for normal socket readiness

File I/O

  • Many operations use libuv thread pool
  • Filesystem APIs complete off the event loop thread
  • Callbacks return when worker operation finishes
  • Can be delayed by thread pool saturation
Two async paths
HTTP socket ready
  -> OS notifies libuv
  -> poll callback
  -> JS handler

fs.readFile
  -> libuv thread pool
  -> OS filesystem work
  -> completion callback queued
  -> JS handler

🎤 Interview short

Network I/O is commonly handled through OS-level non-blocking readiness, while many filesystem operations use the libuv thread pool. Both feel asynchronous in JavaScript, but file operations may compete with other thread pool users like crypto, zlib and DNS lookup.

Part G · Concurrency

Worker Threads

Worker threads give Node real parallel JavaScript execution for CPU-heavy work.

19
Worker thread model
Main Thread
|
├── Event Loop
├── V8 Isolate
└── MessagePort
        |
        v
Worker Thread
|
├── Separate Event Loop
├── Separate V8 Isolate
└── MessagePort

Best for

CPU-heavy JavaScript: parsing, image processing, calculations, compression wrappers, heavy transformations.

Communication

Workers communicate using messages, transferable objects or shared memory with SharedArrayBuffer.

Not for

Do not use workers just to make normal database or HTTP I/O async. Node already handles I/O asynchronously.

Worker thread skeleton
const { Worker, isMainThread, parentPort, workerData } = require('node:worker_threads');

if (isMainThread) {
  const worker = new Worker(__filename, { workerData: 42 });
  worker.on('message', console.log);
} else {
  parentPort.postMessage(workerData * 2);
}
Remember: Worker threads are for CPU parallelism, not for fixing normal async I/O.
Part G · Concurrency

Cluster vs Child Process vs Worker Threads

Worker threads share a process. Cluster and child processes create separate processes.

20

UnitThread inside same processMultiple Node processesSeparate process
MemorySeparate isolate, can share memory deliberatelySeparate memorySeparate memory

Worker Threads

Use when one Node process needs parallel CPU work without creating a full process for each job.

Cluster

Use to run multiple Node server processes, often one per CPU core, behind shared server port/load distribution.

Child Process

Use to run shell commands, other programs, or isolated Node scripts.

🎤 Interview short

Worker threads are best for CPU-heavy JavaScript inside the same process. Cluster creates multiple Node processes to scale a server across CPU cores. Child processes are more general and useful when we need to run external commands or isolate work at the process level.

Part H · Data Flow

Streams & Backpressure

Streams are Node's memory-safe way to process large data piece by piece instead of loading everything at once.

21
Stream types
Streams
|
├── Readable   -> data source
├── Writable   -> data destination
├── Duplex     -> both readable and writable
└── Transform  -> modify data while passing through
Pipeline example
const fs = require('node:fs');
const { pipeline } = require('node:stream/promises');
const zlib = require('node:zlib');

async function compress() {
  await pipeline(
    fs.createReadStream('input.log'),
    zlib.createGzip(),
    fs.createWriteStream('input.log.gz')
  );
}

Backpressure

When the consumer is slower than the producer, Node signals the producer to slow down.

highWaterMark

The buffer threshold that helps control how much data accumulates.

drain

Writable streams emit drain when it is safe to write more after pressure.

Remember: Backpressure protects memory.

Part I · Production

Performance Debugging & Event Loop Lag

In Node, one blocked event loop can make every user request slower.

23

Blocking causes

CPU loops, sync filesystem, huge JSON parse/stringify, catastrophic regex, large compression, heavy crypto.

Event loop lag

The delay between when a callback should run and when the event loop actually gets to run it.

Fixes

Chunk work, use streams, move CPU work to workers, avoid sync APIs, add caching carefully, optimize algorithms.

Simple event loop lag measurement
let last = Date.now();
setInterval(() => {
  const now = Date.now();
  const lag = now - last - 100;
  last = now;
  console.log('event loop lag:', lag, 'ms');
}, 100);
Debugging stack
Production issue
|
├── Check latency and event loop lag
├── CPU profile / flamegraph
├── Heap snapshot for memory leaks
├── Logs and traces
├── Thread pool pressure
└── Slow external dependencies
Remember: If the event loop is blocked, Node cannot accept/process callbacks quickly even if the server has network connections waiting.

Part J · Trainer

Interview Shorts, Output Practice & Quiz

This final chapter turns the architecture into interview answers and output prediction practice.

25

🎤 Explain Node.js architecture in 60 seconds

Node.js is a JavaScript runtime built on V8. V8 executes JS, handles the call stack, heap, optimization and garbage collection. Node adds core APIs like fs, http, crypto, streams, timers and process. Many of those APIs go through native C++ bindings into libuv, OpenSSL, zlib or OS APIs. libuv provides the event loop, timers, poll phase, check phase and worker pool. JavaScript runs on the main thread by default, but Node can offload I/O to the OS, file/crypto/zlib/DNS work to the libuv thread pool, and CPU-heavy JavaScript to worker threads.
