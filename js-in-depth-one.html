Chapter 1

Core Mental Model

The cleanest senior-level explanation starts with one distinction: JavaScript execution is single-threaded, but the browser platform around it is multi-threaded and multi-process.

02
✅

One-line model: JavaScript runs on the call stack; async work is delegated to browser capabilities; completed callbacks are queued; the event loop schedules them back onto the call stack when it is safe.

High-level browser async flow
Call StackJS runs here
→
Web APIsTimers, network, DOM
→
QueuesTask / microtask / animation
→
Event Loopselects next work
→
Call StackJS engine executes

What is single-threaded?

The JavaScript engine executes normal JavaScript on one main call stack. It cannot run two pieces of synchronous JavaScript at the same exact moment on the same thread.

What is multi-threaded?

The browser uses other threads and processes for networking, timers, compositing, rasterization, workers, GPU work, and more.

🎤 Senior interview answer

JavaScript itself runs on a single call stack, usually on the renderer main thread in a browser. But the browser is not single-threaded. It has networking, timer, compositor, raster, GPU, and worker infrastructure. When we call an async API like fetch, JavaScript starts the operation and returns. The actual network work happens outside the call stack. When the result is ready, the browser queues a continuation, usually a promise microtask, and the event loop later lets the JS engine execute that continuation on the call stack.

Chapter 2

JavaScript Engine

The JavaScript engine is responsible for parsing, compiling, optimizing, executing JavaScript, managing memory, and running promise jobs.

03
JavaScript engine architecture
JavaScript Engine
|
├── Parser / Compiler / Optimizer
├── Call Stack
├── Heap
├── Execution Context
├── Lexical Environment
├── Garbage Collector
└── Promise Job Handling
📚

Call Stack

Tracks active function calls. Synchronous JavaScript executes here. A long-running stack blocks user interaction and rendering.

🧺

Heap

Memory area where objects, arrays, functions and closures are allocated. The garbage collector reclaims unreachable memory.

🧾

Execution Context

Contains variable environment, lexical environment, scope chain, this, and information needed to execute code.

🔒

Lexical Environment

Stores bindings and outer references. This is why closures remember variables from their creation scope.

♻️

Garbage Collector

Finds memory that is no longer reachable and frees it. It is automatic, but leaks can still occur through retained references.

⚡

Promise Jobs

Promise reactions are queued as microtasks/jobs. They run after the current synchronous task completes and before the next task.

Call stack example

  console.log("inside c");
}

function b() {
  c();
}

function a() {
  b();
}

a();

a() enters stack

The JS engine starts executing function a.

2
b() then c() enter stack

Each nested call adds a new stack frame.

3
Frames pop after completion

c returns first, then b, then a.

🎤 Senior phrasing

The event loop does not directly “execute” JavaScript. It coordinates when work can enter the call stack. The JS engine executes the selected callback. After a task completes, the browser drains the microtask queue. Then, if it is time to render, animation callbacks and the rendering pipeline can run before the event loop moves to another task.

Queue priority example
setTimeout(() => console.log("task"), 0);

Promise.resolve().then(() => console.log("microtask"));

queueMicrotask(() => console.log("queued microtask"));

console.log("sync");
➡️

Output: sync, then microtask, then queued microtask, then task.

Style

Browser calculates which CSS rules apply and computes final values.

Layout

Browser calculates element sizes and positions. Layout can be expensive if repeatedly forced.

Paint & Composite

Browser paints visual content and composites layers to the screen, often assisted by compositor/GPU infrastructure.

Rendering order example
setTimeout(() => {
  console.log("timer");
}, 0);

Promise.resolve().then(() => {
  console.log("promise");
});

requestAnimationFrame(() => {
  console.log("animation");
});

console.log("sync");
🧭

Common simplified output: sync → promise → animation → timer, assuming the browser has a rendering opportunity before the timer task. In real browsers, timing details can vary, but the interview concept is: synchronous code first, microtasks before the next task, and requestAnimationFrame before paint.

Why microtasks can block rendering

The browser drains microtasks until the microtask queue is empty. If a microtask keeps scheduling another microtask, rendering may be delayed. That means an infinite promise chain can freeze the UI even though each individual callback seems small.

Chapter 8

Async/Await Internals

async/await is syntax over promises. It makes asynchronous code look synchronous, but it does not block the main thread.

09
Async function example
async function test() {
  console.log(1);

  await fetch("/api");

  console.log(2);
}

test();
console.log(3);
1
test() is called

The function starts synchronously and logs 1.

2
await fetch() starts async work

The fetch request is initiated through browser networking. The function suspends and returns a promise to the caller.

3
Outer code continues

console.log(3) runs because await does not block the thread.

4
Continuation becomes a microtask

When the awaited promise settles, the rest of the async function is queued as a promise job/microtask.

5
console.log(2) runs later

The JS engine resumes the function on the call stack when the microtask is processed.

✅

Output idea: 1, 3, then later 2.

🎤 Senior answer

When an async function hits await, it pauses that function, not the JavaScript thread. The async function returns a promise. Once the awaited value settles, the continuation after await is scheduled as a microtask, so it runs after the current synchronous work and before the next task.

Chapter 9

Output Prediction Practice

Interviewers often test async knowledge by asking for output order. Use the stack → microtask → render opportunity → task model.

10
Example 1
console.log("A");

setTimeout(() => console.log("B"), 0);

Promise.resolve().then(() => console.log("C"));

console.log("D");

Answer

A → D → C → B

Synchronous logs run first. Promise callback is a microtask. Timer callback is a task.

Example 2
async function f() {
  console.log("1");
  await Promise.resolve();
  console.log("2");
}

console.log("3");
f();
console.log("4");

Answer

3 → 1 → 4 → 2

The async function starts synchronously. The continuation after await runs as a microtask.

Example 3
setTimeout(() => console.log("timer 1"), 0);

Promise.resolve().then(() => {
  console.log("promise 1");
  Promise.resolve().then(() => console.log("promise 2"));
});

setTimeout(() => console.log("timer 2"), 0);

console.log("sync");

Answer

sync → promise 1 → promise 2 → timer 1 → timer 2

All microtasks drain before timers. The second promise is added while draining microtasks, so it also runs before timers.

Example 4
button.addEventListener("click", () => {
  console.log("click");

  Promise.resolve().then(() => console.log("promise"));

  setTimeout(() => console.log("timer"), 0);
});

Answer after click

click → promise → timer

The click handler is a task. After the task completes, microtasks run, then the timer task can run later.

🧠

Method: First mark synchronous code. Then mark promise/await microtasks. Then mark timers/events as tasks. Then consider rendering APIs such as requestAnimationFrame.

Chapter 10

Common Misconceptions

These are the mistakes that separate intermediate explanations from senior-level browser runtime explanations.

11

❌ “The event loop executes callbacks.”

Correct: The event loop schedules work. The JS engine executes the callback on the call stack.

❌ “JavaScript async means multi-threaded JS.”

Correct: Async means non-blocking scheduling. Normal JS still runs on one call stack unless you explicitly use workers.

❌ “await blocks the thread.”

Correct: await suspends the async function and schedules the continuation later as a microtask.

❌ “Promises go to the task queue.”

Correct: Promise reactions go to the microtask queue / promise job queue.

❌ “setTimeout(fn, 0) runs immediately.”

Correct: It schedules a task. It runs only after current synchronous work, microtasks, and scheduling constraints.

❌ “The browser is single-threaded.”

Correct: JavaScript execution is single-threaded on the main stack, but the browser uses many threads/processes.

What to say instead of “callback queue”

Say: “The callback may be scheduled into a task queue, microtask queue, animation frame callbacks, or idle callback queue depending on the API and browser scheduling rules.”

🎤 Full senior answer

In the browser, JavaScript normally executes on a single call stack inside the JavaScript engine, usually on the renderer main thread. The engine also manages heap memory, execution contexts, lexical environments, garbage collection and promise jobs.

The browser around it is multi-threaded and multi-process. It provides Web APIs such as timers, fetch, DOM events, observers, workers and rendering APIs. When JavaScript starts async work, the work is delegated to browser infrastructure such as networking, timers or workers. When the async operation completes, the browser schedules a callback or continuation into the correct queue.

The event loop coordinates this scheduling. It picks tasks when the stack is free, drains microtasks after each task, allows animation frame callbacks before paint, then the browser can run style, layout, paint and composite. The event loop schedules work onto the call stack; the JavaScript engine executes it.

Final Preparation

Senior Interview Answers

Use these as polished spoken answers. They are intentionally concise but complete.

16
Explain browser async JavaScript in 60 seconds

JavaScript runs on a single call stack inside the JavaScript engine. The browser around it is multi-threaded and provides Web APIs like timers, fetch, DOM events, workers and rendering APIs. When JS starts async work, the work is delegated to the browser. When it completes, the browser schedules a callback or continuation into a queue. Promise callbacks and await continuations go to the microtask queue. Timers and events usually enter task queues. The event loop coordinates when queued work can go back to the call stack. After each task, microtasks are drained. Then the browser may run animation frame callbacks and rendering steps like style, layout, paint and composite.

Explain fetch internals

fetch starts a network request through the browser's networking layer and returns a promise immediately. JavaScript does not wait for the HTTP request on the call stack. When the network response is ready, the promise settles, and the related .then or await continuation is scheduled as a microtask. The JS engine later runs that continuation when the stack is empty and after the current task finishes.

Explain microtask vs task

A task is a larger unit of scheduled work such as a timer callback, user event, or message event. A microtask is higher-priority follow-up work such as a promise reaction, await continuation or queueMicrotask. After the current script or task finishes, the browser drains all microtasks before moving to the next task.

Explain requestAnimationFrame

requestAnimationFrame schedules a callback before the browser's next paint. It is ideal for visual updates because the browser can align the callback with the display refresh cycle. It is not the same as a promise microtask or timer task; it belongs to the rendering side of the browser's scheduling model.

Explain why the UI freezes

The UI freezes when the main thread is blocked. Long synchronous JavaScript, expensive layout, heavy paint work, or an endless microtask chain can prevent the browser from handling input and rendering frames. The fix is to break work into chunks, avoid forced synchronous layout, use workers for CPU-heavy work, and schedule visual changes with rendering-aware APIs.

Explain browser vs Node event loop

The browser event loop integrates JavaScript execution with DOM events and rendering. It has tasks, microtasks, animation frame callbacks, idle callbacks and the rendering pipeline. Node.js does not have DOM rendering; it uses libuv with phases such as timers, poll and check, plus special queues like process.nextTick and promise microtasks.

🏁

Final target: In an interview, do not merely say “callback queue and event loop.” Talk about the JavaScript engine, browser APIs, multi-threaded browser runtime, task/microtask scheduling, and rendering pipeline.
